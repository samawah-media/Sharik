# دليل الاختبار اليدوي للمالك - Spec 015

الحالة: دليل تجربة يدوية للمالك على بيئة Preview/UAT غير الإنتاجية فقط.

استخدم هذا الملف لتجربة منصة سماوة خطوة بخطوة من كل منظور أساسي: الإدارة، مدير الحساب، الفريق المسند، العميل المشاهد، والعميل المعتمد. هذا الدليل لا يعني إطلاق Production، ولا يسمح بدعوة عميل خارجي حقيقي الآن.

## 1. قبل أن تبدأ

### البيئة الصحيحة

- استخدم رابط Preview/UAT المعتمد فقط.
- لا تستخدم Production.
- لا تستخدم روابط Preview قديمة أو روابط عشوائية.
- لا تختبر بحساب عميل خارجي حقيقي الآن.
- لا تنسخ كلمات مرور أو إيميلات أو توكنات أو روابط خاصة داخل Git أو الصور أو المحادثات العامة أو ملفات الأدلة.
- خذ الرابط وحسابات الاختبار من المصدر المحلي الآمن فقط، وليس من ملفات موثقة داخل Git.

المصدر المحلي الآمن:

```text
.env.s015-team-uat.local
```

استخدم هذه القيم كمدخلات خاصة لك فقط:

```text
S015_UAT_BASE_URL
S015_UAT_ALLOWED_HOSTNAME
```

### طريقة فتح المتصفح

استخدم نافذة خاصة أو بروفايل جديد لكل دور.

الطريقة المقترحة:

1. افتح نافذة خاصة.
2. افتح صفحة تسجيل الدخول في Preview/UAT المعتمد.
3. سجل الدخول بدور واحد فقط.
4. نفذ قائمة اختبار هذا الدور.
5. سجل الخروج.
6. أغلق النافذة الخاصة.
7. كرر نفس الخطوات مع الدور التالي.

### متى توقف الاختبار فورًا؟

أوقف الاختبار وسجل عيبًا إذا حدث أي مما يلي:

- فشل تسجيل الدخول لحساب UAT معتمد.
- صفحة مهمة ترجع 404 بعد تسجيل الدخول.
- عميل يرى بيانات عميل آخر.
- عميل يرى تعليقات داخلية أو مهام داخلية أو ملاحظات جودة أو ملفات داخلية.
- العميل المشاهد يستطيع الاعتماد أو طلب تعديل أو التعليق أو الرفع.
- عضو الفريق يستطيع التعميد الداخلي أو الإرسال للعميل أو التسليم النهائي بدون صلاحية.
- مخرج يظهر للعميل قبل التعميد الداخلي والإرسال الرسمي.
- SLA يستمر في احتساب التأخير على سماوة أثناء انتظار العميل.
- الواجهة تعرض UUID أو أسماء تقنية أو حالات خام أو عربي تالف.
- أي عملية حفظ تظهر ناجحة ثم تختفي بعد تحديث الصفحة.

## 3. تجربة الإدارة

الهدف: التأكد أن الإدارة تفهم حالة التشغيل بالكامل وتتحكم في سير العمل بدون بيانات اختبار مربكة.

الدور:

```text
samawah_admin أو حساب إدارة معتمد للاختبار
```

الخطوات:

1. سجل الدخول.
2. افتح `/work`.
3. تأكد أن الصفحة تعمل بدون 404 أو دوران تسجيل دخول.
4. تأكد أن التنقل مختصر ومفهوم.
5. تأكد أن الصفحة تعرض عمل Hadna وGlass فقط، بدون Alpha/Beta أو بيانات negative-control.
6. تأكد أن كل كرت يعرض اسم العميل، المنصة أو القناة، نوع المخرج، الحالة، المالك، الموعد، SLA، والخطوة التالية.
7. افتح تفاصيل مخرج.
8. راجع أقسام المحتوى، النسخ، الملفات، التعليقات، الجودة، والنشاط.
9. تأكد أن التعليقات الداخلية وملاحظات الجودة تظهر للإدارة.
10. تأكد أنه لا تظهر UUID أو حالات تقنية أو عربي تالف.
11. حدث الصفحة وتأكد أن البيانات كما هي.

النتيجة المتوقعة:

- الإدارة تفهم ماذا يحدث، من المسؤول، ما الذي ينتظر قرارًا، وما الخطوة التالية.
- الإدارة ترى التفاصيل الداخلية المطلوبة للتشغيل.
- لا يتم تطبيق قيود العميل بالخطأ على الإدارة.

سجل النتيجة:

```text
تجربة الإدارة: PASS / FAIL
ملاحظات:
عيوب:
```

## 4. تجربة مدير الحساب

الهدف: التأكد أن مدير الحساب يستطيع تنسيق العمل بدون تجاوز صلاحيات الاعتماد أو التسليم.

الدور:

```text
account_manager
```

الخطوات:

1. سجل الدخول.
2. افتح `/work`.
3. تأكد أن أعمال العملاء المسندة أو المصرح بها تظهر.
4. تأكد أن كروت المخرجات واضحة وغير مكررة.
5. افتح تفاصيل مخرج.
6. تأكد أن مدير الحساب يرى سياق العمل المطلوب للتنسيق.
7. جرّب تحديث مهمة أو checklist إذا كان ذلك متاحًا.
8. تأكد أنه لا يستطيع التعميد الداخلي إلا إذا كانت الصلاحية مصرحًا بها صراحة.
9. تأكد أنه لا يستطيع الاعتماد كعميل.
10. تأكد أنه لا يستطيع التسليم النهائي إذا لم يسمح workflow بذلك.
11. حدث الصفحة وتأكد أن التحديثات المسموحة بقيت محفوظة.

النتيجة المتوقعة:

- مدير الحساب يستطيع التنسيق والمتابعة.
- لا يستطيع تجاوز التعميد الداخلي أو اعتماد العميل أو عزل العملاء أو التسليم النهائي.

سجل النتيجة:

```text
تجربة مدير الحساب: PASS / FAIL
ملاحظات:
عيوب:
```

## 5. تجربة الفريق المسند

الهدف: التأكد أن الكاتب أو المصمم يجد عمله بسهولة، ويحدث مهامه، ويرفع نسخة، ويتعاون داخليًا بدون رؤية أعمال غير مصرح بها.

الأدوار:

```text
content_writer
designer
```

نفذ هذه التجربة مرة للكاتب ومرة للمصمم إذا كانت الحسابات متاحة.

الخطوات:

1. سجل الدخول ككاتب أو مصمم مسند.
2. افتح `/work`.
3. تأكد أن أول عرض مفيد هو "مهامي" أو قائمة الأعمال المسندة.
4. تأكد أن الأعمال الظاهرة هي المسندة أو المصرح بها فقط.
5. افتح مخرجًا مسندًا.
6. راجع المحتوى، التعليمات، التعليقات، الملفات، ملاحظات الجودة، والنشاط.
7. أنشئ أو حدث مهمة مسندة إذا كان هذا متاحًا.
8. غيّر حالة مهمتك فقط إذا كان هذا متاحًا.
9. ارفع نسخة عمل جديدة إذا كانت حالة المخرج تسمح بذلك.
10. أضف تعليقًا داخليًا إذا كان متاحًا.
11. حدث الصفحة وتأكد أن التحديث محفوظ.
12. حاول الوصول إلى عمل غير مسند لك.
13. تأكد أن العمل غير المصرح به مخفي أو مرفوض.

النتيجة المتوقعة:

- عضو الفريق يستطيع العمل داخل صلاحياته.
- لا يرى أعمال عملاء أو مخرجات غير مصرح بها.
- لا يستطيع التعميد الداخلي أو الإرسال للعميل أو الاعتماد كعميل أو التسليم النهائي.
- التعليقات الداخلية تبقى داخلية.

سجل النتيجة:

```text
تجربة الفريق المسند: PASS / FAIL
الدور المختبر:
ملاحظات:
عيوب:
```

## 6. تجربة العميل المشاهد

الهدف: التأكد أن العميل المشاهد يرى المعلومات الآمنة فقط، وبدون أي قدرة على التعديل.

الدور:

```text
client_viewer
```

الخطوات:

1. سجل الدخول.
2. افتح `/client`.
3. تأكد أن الصفحة الرئيسية للعميل بسيطة ومفهومة.
4. افتح `/client/pending`.
5. تأكد أن العناصر المنتظرة تعرض النسخ المعتمدة داخليًا والمرسلة للعميل فقط.
6. تأكد أن النص أو الوصف أو fallback واضح ومفهوم.
7. تأكد أنه لا توجد أزرار اعتماد أو طلب تعديل.
8. تأكد أنه لا يوجد حقل تعليق.
9. تأكد أنه لا يوجد زر رفع ملف.
10. افتح `/client/files`.
11. تأكد أن الملفات الظاهرة هي النهائية أو المصرح بها للعميل فقط.
12. تأكد أن الملفات الداخلية والتعليقات الداخلية والمهام وملاحظات الجودة والنشاط الداخلي مخفية.
13. حدث الصفحة وتأكد أن وضع القراءة فقط ثابت.

النتيجة المتوقعة:

- العميل المشاهد يفهم الحالة ويرى المواد المسموحة فقط.
- لا يستطيع تعديل أي شيء.
- لا يرى عمل سماوة الداخلي.

سجل النتيجة:

```text
تجربة العميل المشاهد: PASS / FAIL
ملاحظات:
عيوب:
```

## 7. تجربة العميل المعتمد

الهدف: التأكد أن العميل المعتمد يراجع النسخة الحالية الصحيحة فقط، ويتخذ القرار عندما يكون مصرحًا له.

الدور:

```text
client_approver
```

مرحلة الفحص بدون تعديل:

1. سجل الدخول.
2. افتح `/client/pending`.
3. تأكد أن العنصر المنتظر واضح ومفهوم.
4. تأكد أن النسخة المعروضة هي النسخة الحالية المرسلة للعميل.
5. تأكد أن أزرار الاعتماد وطلب التعديل ظاهرة.
6. تأكد أن التعليقات الداخلية والمهام وملاحظات الجودة والملفات الداخلية والنسخ القديمة و UUID مخفية.
7. لا تضغط اعتماد أو طلب تعديل في أول مرور إلا إذا قررت بدء تجربة تعديل حقيقية داخل UAT.

تجربة القرار، فقط إذا وافق المالك على mutation داخل UAT:

1. اختر عنصر UAT واحدًا فقط.
2. سجل نوع العنصر أو فئته بدون نسخ محتواه الخاص.
3. اضغط طلب تعديل.
4. اكتب تعليقًا آمنًا مثل: `UAT change request`.
5. تأكد أن العنصر عاد للحالة الصحيحة في workflow.
6. سجل دخولك كإدارة وتأكد أن SLA خرج من انتظار العميل واستأنف العمل.
7. اجعل الفريق يرفع نسخة معدلة.
8. اجعل الإدارة تعتمد داخليًا وترسل للعميل.
9. سجل الدخول مرة أخرى كعميل معتمد.
10. اعتمد النسخة المعدلة.
11. تأكد أن سجل النشاط أو audit يظهر القرار.

النتيجة المتوقعة:

- العميل المعتمد يقرر فقط على النسخة الحالية المرسلة له.
- القرار ينشئ أثر audit.
- SLA يتوقف أثناء انتظار العميل ويستأنف عند طلب التعديل.

سجل النتيجة:

```text
تجربة العميل المعتمد: PASS / FAIL
نوع التجربة: فحص فقط / تجربة قرار
ملاحظات:
عيوب:
```

## 8. تجربة إنشاء عميل وعقد وباقة ومخرج جديد

الهدف: التأكد أن المنصة تستطيع استقبال عميل UAT جديد من البداية حتى إنشاء مخرج.

استخدم أسماء اختبار فقط، ولا تدخل أسرار عميل حقيقي.

أسماء مقترحة:

```text
العميل: UAT Owner Trial Client
العقد: UAT Owner Trial Contract
الباقة: UAT Owner Trial Package
المخرج: UAT Owner Trial Content Post
```

خطوات الإدارة:

1. سجل الدخول كإدارة.
2. افتح `/clients`.
3. اضغط إنشاء عميل جديد أو افتح `/clients/new`.
4. أنشئ عميل UAT.
5. افتح صفحة العميل الجديد.
6. افتح العقود.
7. أنشئ عقدًا جديدًا.
8. افتح باقات العقد.
9. أنشئ باقة إذا كانت الواجهة تتطلب ذلك.
10. افتح المخرجات.
11. أنشئ مخرجًا جديدًا.
12. عيّن مالكًا أو مساهمين إذا كانت الواجهة تسمح.
13. تأكد أن المخرج يظهر في `/work`.
14. تأكد أن الفريق المسند يراه فقط إذا كان مسندًا له.
15. تأكد أن العميل لا يراه قبل التعميد الداخلي والإرسال للعميل.

النتيجة المتوقعة:

- إنشاء العميل والعقد والمخرج يعمل بدون كسر عزل tenant/client.
- المخرج يبدأ في الحالة الصحيحة.
- المخرج مخفي عن العميل حتى يتم تعميده داخليًا وإرساله.

سجل النتيجة:

```text
إنشاء عميل/عقد/مخرج جديد: PASS / FAIL
اسم عميل UAT المستخدم:
ملاحظات:
عيوب:
```

## 9. تجربة المحتوى والنسخ والتعليقات والملفات

الهدف: التأكد أن العمل الحقيقي يتحرك داخل المنصة.

خطوات الفريق المسند:

1. سجل الدخول ككاتب أو مصمم مسند.
2. افتح `/work`.
3. افتح مخرج UAT.
4. أضف أو حدث محتوى العمل إذا كان ذلك متاحًا.
5. ارفع نسخة جديدة.
6. أضف تعليقًا داخليًا.
7. ارفع ملف UAT آمن إذا كان الرفع مفعّلًا.
8. حدث الصفحة وتأكد أن النسخة والتعليق والملف باقية.

خطوات الإدارة:

1. سجل الدخول كإدارة.
2. افتح نفس المخرج.
3. تأكد أن النسخة المرفوعة ظاهرة.
4. أضف ملاحظة جودة إذا كانت متاحة.
5. اطلب تعديلًا داخليًا، ثم تأكد أن الفريق يراه.
6. بعد التعديل، اعتمد داخليًا.
7. أرسل النسخة المعتمدة للعميل.

خطوات العميل:

1. سجل الدخول كعميل مشاهد.
2. تأكد أنه يرى النسخة المرسلة فقط وبدون أزرار قرار أو كتابة.
3. سجل الدخول كعميل معتمد.
4. تأكد أن أزرار القرار تظهر.
5. نفذ فحصًا فقط أو تجربة قرار حسب موافقة المالك.

النتيجة المتوقعة:

- النسخ دقيقة ومستمرة بعد التحديث.
- التعليقات تحترم قواعد الرؤية.
- الملفات تحترم قواعد الرؤية.
- العميل لا يرى المواد الداخلية.
- التحولات الحساسة تسجل audit.

سجل النتيجة:

```text
تجربة المحتوى/النسخ/التعليقات/الملفات: PASS / FAIL
ملاحظات:
عيوب:
```

## 10. تجربة اللوحة وسير العمل

الهدف: التأكد أن Board مفيدة، ولا تسمح بتجاوز workflow.

الخطوات:

1. سجل الدخول كإدارة.
2. افتح لوحة مخرجات عميل، غالبًا عبر `/clients/[clientId]/deliverables/board`.
3. تأكد أن الأعمدة مفهومة.
4. حرّك أو حدّث مخرج UAT فقط إذا كان ذلك متاحًا.
5. تأكد أن التحولات غير القانونية مرفوضة.
6. تأكد أن التحولات القانونية تبقى بعد تحديث الصفحة.
7. سجل الدخول كفريق مسند.
8. تأكد أن الفريق يستخدم فقط الأفعال المسموحة له.
9. سجل الدخول كعميل.
10. تأكد أن العميل لا يستطيع فتح اللوحة الداخلية.

النتيجة المتوقعة:

- اللوحة مفيدة للتشغيل.
- اللوحة لا تتجاوز التعميد الداخلي أو اعتماد العميل أو SLA أو audit أو الصلاحيات.

سجل النتيجة:

```text
تجربة Board/workflow: PASS / FAIL
ملاحظات:
عيوب:
```

## 11. تجربة الهاتف و RTL

الهدف: التأكد أن نفس التجربة تعمل على شاشة صغيرة وبالعربية.

الخطوات:

1. افتح Preview/UAT على الهاتف أو محاكاة الهاتف في المتصفح.
2. سجل الدخول كإدارة وافتح `/work`.
3. تأكد أنه لا يوجد تمرير أفقي غير طبيعي.
4. افتح تفاصيل مخرج أو drawer.
5. تأكد أن الأزرار مناسبة وقابلة للضغط.
6. سجل الدخول كعميل معتمد.
7. افتح `/client/pending`.
8. تأكد أن النص العربي والكروت وأزرار الاعتماد وطلب التعديل تظهر بشكل صحيح.
9. سجل الدخول كعميل مشاهد.
10. تأكد أن المشاهد يبقى قراءة فقط على الهاتف.

النتيجة المتوقعة:

- لا يوجد كسر في layout الهاتف.
- النص العربي واضح.
- الأزرار قابلة للاستخدام باللمس.
- الصلاحيات لا تتغير على الهاتف.

سجل النتيجة:

```text
تجربة الهاتف و RTL: PASS / FAIL
ملاحظات:
عيوب:
```

## 12. نموذج تسجيل العيوب

استخدم هذا النموذج لكل عيب.

```text
ID:
الخطورة: P0 / P1 / P2 / P3
الدور:
المسار:
workflow:
خطوات إعادة المشكلة:
المتوقع:
الفعلي:
أثر الأمان/البيانات:

```

دليل الخطورة:

- P0: تسريب بيانات، أثر على Production، تلف مدمر، أو كشف سر.
- P1: يمنع UAT المطلوب، يتجاوز صلاحية، يتجاوز اعتماد، يكشف داخلي للعميل، أو يمنع دخول حساب معتمد.
- P2: مشكلة مهمة في UX أو workflow ولها حل مؤقت.
- P3: تحسين صياغة أو شكل أو تفصيلة غير مانعة.

## 13. قرار المالك النهائي

لا تسجل PASS إلا إذا تحقق كل ما يلي:

- تجربة الإدارة ناجحة.
- تجربة مدير الحساب ناجحة.
- تجربة الفريق المسند ناجحة.
- تجربة العميل المشاهد ناجحة.
- تجربة العميل المعتمد ناجحة.
- لا يوجد P0/P1 مفتوح.
- كل P2 له قرار واضح.
- لم يحدث أي إجراء على Production.
- لم تتم دعوة عميل خارجي حقيقي.
- لم يتم نسخ أي أسرار أو بيانات خاصة داخل Git أو الأدلة.

نموذج القرار النهائي:

```text
قرار UAT اليدوي من المالك: PASS / HOLD
التاريخ:
السبب:
P0/P1 المفتوحة:
P2 المفتوحة مع القرار:
الخطوة التالية المعتمدة:
```

إذا كانت النتيجة PASS:

```text
ننتقل إلى تجربة فريق سماوة الداخلية على Preview/UAT غير الإنتاجية فقط.
لا يوجد merge أو Production أو دعوة عميل خارجي حقيقي إلا بعد بوابة إطلاق منفصلة.
```

إذا كانت النتيجة HOLD:

```text
سجل العيوب في defect register الخاص بـ Spec 015.
أصلح P0/P1 أولًا.
أعد اختبار المسارات المتأثرة ثم نفذ مرورًا مختصرًا على باقي الأدوار.
```
